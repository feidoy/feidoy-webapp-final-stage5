Servers:

The Servers purpose: Respond to HTTP requests.


1. static content: prewritten file (image,)


2. dynamic content: response is built on the fly, by web applications. Web apps are programs that generate content. Web Apps live on webservers, speaks http, and generates content that browsers request.

http = is the protocol used on the world wide web. Hyper Text Transfer Protocol, used to talk to the servers.

GET/foo http/1.1 - request line
3 main parts to request line =

1. method
2. path
3. version

method (get, post,etc) , path (ex /foo - actuall document requesting from server) - used for making request, last is version (version of HTTP).
for path, no need to use fragment and beyond (#)...

GET and POST:

How GET requests and POST requests differ.

GET - for fetching docs

1. parameters in URL
2. used for fetching documents
3. maximum URL length
4. Ok to cache
5. shouldn't change the server

POST - used for making updates to server (can be destructive in nature)

1. parameters in body
2. used for updating data
3. no max length
4. not ok to cache (industry standard, don't cache post requests)
5. ok to change the server


Validation:

Make sure to always validate your parameters. Servers can receive junk, and it can take up resources to process the junk. Validating parameters prevents this.


Always verify data on the backend. Verify what the users enter and complain if the data is bad.


You <---------------> Server


1. GET form --> # make a get request to server

2. <--- form data # server sends form data back to user

3. Post form -- > # make a post request to server with data

4. Server will run validation functions. If data is good, server will say thanks! If data is bad, server will respond with form data again, send error message to re-enter values. Circles back to #3 Post form.

Verify if the users input, on error render the form again, and include an error message. Validation improves user experience by letting the user know when the data input is bad. When the data is bad, it will send an an error message and reload the form, and if it's good, it should have a redirect message indicating a successful input.

A redirect helps so that reloading a page doesn't resubmit the form, and so we can also have distinct pages for forms, and success pages. You can make a "Thanks handler", and a new "Thanks.html" URL and also initialize a redirect to the /thanks URL. You can use self.redirect("/path"), then make sure to add to the mapping.


HTML Escaping:

HTML escaping allows you to escape or leave out certain characters like "", < >, & - so the forms don't get busted up.

For " use --> &quot;
For > use --> &gt;
For < use --> &lt;
For & use --> &mp;

Helpful TIPS -

1. Always automatically escape variables when possible - not every template language will do so, but always check.
2. minimize code in templates - makes it difficult to modify, debug.
3. minimize html in code - put it in a template.

HTTP Responses and Making Requests

Making Requests -
Headers = Name: value (name, semicolon, value)
When you make a http request, all of the below (1-3) is sent at once. Request line, Host, user-agent.


(1) GET /foo?p=1 HTTP/1.1 <-- request line

(2) host: www.example.com <-- which website we are requesting the document for, required for http/1.1. one server can host many websites, so we need to say what website we are requesting the document for.

(3) user-agent:chrome v.17 <--- this header is describing what or who is making the request... in this case its the browser, knows what type of machine is making the request.


################################ ### Http responses ######### ################################

browser sends request, server responds with a response.

basic http response looks like:
status line - HTTP/1.1 200 OK

Status Codes:
200 = 200 is ok
300 = is found/somewhere else
404 = not found
500 = server error

Status code is from 1,2 everything ok,3 document is located somewhere else - we need to do something technical to find the document,4 error on browser side,5 error on server side, broke handling this request.

http response headers:
1. Date - here all the time, when the request happens

2. Server: name and version of the server that's handling the response... NEVER INCLUDE THIS, don't give free information to a would be hacker.

3. Content-type : type of document being returned. tells browser what to display, how to display. (if image, shows image, etc.)

4. Content-length : How long the document is. tells when browser is done sending data. Often included, but not required. A way to tell browser is done sending data.
